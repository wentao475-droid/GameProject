# 星块爆破 Web 版产品设计文档

## 1. 文档定位

本设计文档用于定义 `星块爆破` Web 版从“可玩 MVP”升级为“可上线验证的完整游戏产品”的方案，重点覆盖以下两步：

1. 先把当前 Web 版打磨成更完整的游戏产品。
2. 补齐本地存档、设置、开始流程、结算流程。

本文档面向产品、设计、前端开发与后续 iOS 封装阶段，目标是让当前 Web 版本具备更完整的单局体验、更稳定的留存能力，以及更接近原生休闲游戏的产品形态。

---

## 2. 当前版本现状

### 2.1 已有能力

基于当前仓库实现，Web 版已经具备以下能力：

- 单页可玩游戏入口。
- `10 x 10` 棋盘与 `5` 色随机方块。
- 相邻同色块识别与合法消除。
- 分数计算规则：`n * n * 5`。
- 消除后下落与空列左移。
- 无效点击反馈。
- 无可用步时结束。
- 移动端竖屏优先布局。
- 右上角分数展示。
- 基础结算弹层。
- 残局自动清盘。

### 2.2 当前不足

虽然已经“能玩”，但还没有达到“完整游戏产品”的标准，主要缺口如下：

- 缺少开始页，用户进入即开局，缺乏仪式感与新手引导入口。
- 缺少本地存档，刷新页面后最高分、偏好设置、最近战绩全部丢失。
- 缺少设置页或设置面板，用户无法控制音效、震动、动画强度、主题等偏好。
- 缺少完整结算页结构，当前结算偏功能性，缺少战绩对比与下一步动作引导。
- 缺少局外状态管理，无法形成“开始 -> 游戏中 -> 结算 -> 再来一局”的稳定闭环。
- 缺少长期扩展位，如后续接入 iOS 包壳时需要的本地持久化、用户偏好、版本信息与隐私入口。

### 2.3 产品判断

因此，下一阶段不应继续堆叠玩法复杂度，而应优先完成“产品壳层”的建设，让当前 Web 版从“演示原型”升级为“可持续迭代的正式前端产品”。

---

## 3. 产品目标

### 3.1 核心目标

在不增加复杂商业化和联网系统的前提下，把当前 Web 版升级为一个完整的单机休闲游戏产品，使其满足以下条件：

- 首次进入有明确开始入口。
- 单局过程有完整状态反馈。
- 游戏结束后有完整结算与再来一局路径。
- 用户偏好与关键成绩可在本地保存。
- UI 形态更接近移动端原生休闲游戏。
- 为后续封装 iOS App 做好产品层准备。

### 3.2 阶段目标

#### 阶段一：完整游戏产品化

重点解决：

- 首页感与开始流程。
- 游戏中的信息层级。
- 更完整的结算反馈。
- 更强的“再玩一局”转化。

#### 阶段二：局外系统补齐

重点解决：

- 本地持久化。
- 设置项与偏好项。
- 历史最佳记录。
- 最近战绩和可恢复的用户习惯。

## 4. 目标用户

### 4.1 主要用户

- 使用手机浏览器或未来 iOS App 的轻度休闲玩家。
- 喜欢单手、短时、即时反馈类消除玩法的用户。
- 希望在碎片时间快速完成 1 到 3 局的用户。

### 4.2 使用场景

- 通勤、排队、午休、睡前等碎片时间。
- 单手握持手机、竖屏浏览。
- 静音环境下也能完成完整体验。

### 4.3 用户需求

- 打开后能快速进入游戏。
- 分数和反馈清晰。
- 失败后有继续玩下去的动力。
- 偏好设置不会每次重置。
- 最高分能被记录，形成轻度追求。

---

## 5. 产品原则

### 5.1 移动端优先

- 所有主路径以竖屏单手使用为前提。
- 最核心交互区永远是棋盘本身。
- 信息不抢棋盘注意力。

### 5.2 局内轻，局外完整

- 局内只保留必要信息，不堆叠说明文字。
- 局外通过开始页、设置页、结算页承载完整产品信息。

### 5.3 原创休闲感

- 不使用任何与现有知名产品可混淆的名称、视觉、文案和素材。
- 保持原创色彩语言、按钮语言和结算文案。

### 5.4 离线可玩

- 关键功能全部前端本地可运行。
- 不依赖后端，不依赖登录。

## 7. 信息架构

Web 版升级后，产品应由以下几个主层级组成：

### 7.1 开始页

承担功能：

- 品牌标题展示。
- 开始游戏按钮。
- 查看最佳分。
- 查看最近一局成绩摘要。
- 打开设置面板。

### 7.2 游戏页

承担功能：

- 棋盘操作。
- 当前分数展示。
- 剩余方块展示。
- 重开入口。
- 可选的暂停/返回开始页入口。

### 7.3 结算页

承担功能：

- 本局总分。
- 是否刷新最高分。
- 剩余方块数。
- 自动清盘数量。
- 再来一局。
- 返回首页。

### 7.4 设置面板

承担功能：

- 音效开关。
- 震动开关。
- 动画开关或动画强度。
- 重置本地数据。
- 查看版本与隐私说明入口。

## 8. 核心状态机设计

当前产品需要从“只有游戏中状态”升级为“完整局外/局内状态机”。

### 8.1 顶层状态

建议定义以下页面级状态：

- `home`
- `playing`
- `settling`
- `game-over`
- `settings`

### 8.2 状态流转

#### 首次进入

- 打开页面。
- 读取本地存档。
- 展示开始页。
- 用户点击“开始游戏”。
- 进入对局。

#### 正常结束

- 玩家持续消除。
- 棋盘无可用步。
- 如有残局，执行自动清盘动画。
- 生成结算数据。
- 进入结算页。

#### 再来一局

- 用户点击“再来一局”。
- 重置局内状态。
- 保留局外存档。
- 进入新对局。

#### 返回首页

- 用户点击“返回首页”。
- 保留历史记录与设置。
- 回到开始页。

## 9. 功能设计详述

## 9.1 开始页设计

### 目标

让游戏在进入时更像产品而不是直接落在一个棋盘上。

### 必要模块

- 游戏标题。
- 一句极短副标题，可选。
- 主按钮：开始游戏。
- 次按钮：设置。
- 最佳分展示。
- 最近一局成绩摘要。

### 展示建议

- 中上区域展示品牌与标题。
- 中部展示开始按钮。
- 下部展示最佳分与最近战绩卡片。

### 文案建议

- 标题：`星块爆破`
- 主按钮：`开始游戏`
- 设置按钮：`设置`
- 最佳分：`最佳分数`
- 最近战绩：`最近一局`

### 交互规则

- 首次打开默认停留在开始页。
- 若用户刚完成一局并返回首页，应优先展示“最近一局成绩”。

---

## 9.2 游戏页设计

### 目标

保持当前轻量风格，但让游戏中信息更像正式产品。

### 必要模块

- 右上角分数 HUD。
- 剩余方块 HUD。
- 主棋盘。
- 重开按钮。
- 设置入口或暂停入口。

### 建议优化

- 底部增加轻量工具栏，包含：
  - 重开
  - 返回首页
  - 设置

### HUD 设计要求

- 右上角优先展示当前分数。
- 次级信息展示剩余方块。
- 避免加入冗余说明文字。

---

## 9.3 结算页设计

### 目标

把当前“结束弹窗”升级为真正的结算页，承担留存与复玩转化。

### 必要模块

- 本局得分。
- 是否破纪录。
- 最佳分数。
- 剩余方块数。
- 残局自动清盘数。
- 再来一局按钮。
- 返回首页按钮。

### 结算结果结构

建议输出以下字段：

- `score`
- `bestScore`
- `isNewBest`
- `remainingBlocks`
- `autoClearedBlocks`
- `finishedAt`

### 文案建议

- 标题：`本局结束`
- 新纪录提示：`刷新最佳分`
- 主按钮：`再来一局`
- 次按钮：`返回首页`

### 动作优先级

- 主路径必须是“再来一局”。
- 返回首页作为次级动作。

---

## 9.4 本地存档设计

### 目标

让用户的成绩和设置在浏览器本地持久化，支持 App 封装复用。

### 存档方式

使用 `localStorage`。

### 建议存储结构

```ts
type LocalGameProfile = {
  bestScore: number;
  totalRounds: number;
  lastResult: {
    score: number;
    remainingBlocks: number;
    autoClearedBlocks: number;
    finishedAt: string;
  } | null;
  settings: {
    soundEnabled: boolean;
    vibrationEnabled: boolean;
    animationEnabled: boolean;
  };
};
```

### 存档写入时机

- 游戏结束时写入最佳分与最近战绩。
- 设置改动时即时写入。
- 重置数据时清空对应存储键。

### 存档读取时机

- 应用初始化时读取。
- 若存档不存在，写入默认值。

### 默认值建议

- `bestScore = 0`
- `totalRounds = 0`
- `lastResult = null`
- `soundEnabled = true`
- `vibrationEnabled = true`
- `animationEnabled = true`

---

## 9.5 设置功能设计

### 目标

补齐用户偏好系统，为后续 iOS App 版本直接复用。

### 第一版设置项

- 音效开关。
- 震动开关。
- 动画开关。
- 重置本地数据。

### 设置说明

#### 音效开关

- 当前 Web 版可先只保留配置项，不必一次性接入复杂音效系统。
- 若当前阶段不做音效资源，也应预留开关与状态。

#### 震动开关

- Web 环境可根据浏览器支持情况决定是否启用。
- 即使 Web 里暂不实际触发，也要保留设置项，便于未来 iOS / 小程序接入。

#### 动画开关

- 开启时保留当前消除、下落和残局清盘动画。
- 关闭时使用更短或无动画版本，提升弱设备体验。

#### 重置本地数据

- 清除最佳分、最近一局、设置项。
- 需要二次确认。

### 设置入口形式

优先使用底部抽屉面板或居中弹层，不建议跳独立页面。

---

## 9.6 最近战绩设计

### 目标

强化“再玩一局”的心理闭环。

### 展示字段

- 上一局得分。
- 剩余方块数。
- 自动清盘数。
- 时间戳。

### 展示位置

- 首页展示最近一局摘要。
- 结算页展示本局结果并可与最佳分对比。

---

## 9.7 最佳分设计

### 目标

提供轻度目标感和长期留存动力。

### 规则

- 若本局得分大于历史最佳分，则更新最佳分。
- 结算页展示“刷新最佳分”提示。

### 展示位置

- 首页。
- 结算页。
- 可选放入游戏中 HUD 的次级区域，但不应抢当前分数。

## 12. 详细交互流程

## 12.1 首次进入流程

1. 应用启动。
2. 读取本地存档。
3. 若无存档则初始化默认档案。
4. 展示开始页。
5. 用户点击开始游戏。
6. 生成新棋盘并进入游戏页。

## 12.2 完成一局流程

1. 用户操作棋盘直到无合法步。
2. 若仍有残局则执行自动清盘动画。
3. 生成本局战绩。
4. 更新最佳分与最近一局。
5. 持久化到本地。
6. 展示结算页。

## 12.3 修改设置流程

1. 用户打开设置面板。
2. 切换音效、震动或动画选项。
3. 即时写入本地。
4. 返回当前页面。

## 12.4 重置本地数据流程

1. 用户进入设置。
2. 点击重置本地数据。
3. 二次确认。
4. 清除本地档案。
5. 恢复默认设置和零分状态。

## 17. 验收标准

### 17.1 产品完整性

- 用户首次进入能看到开始页。
- 用户可从开始页进入游戏。
- 用户结束后能进入完整结算页。
- 用户可以从结算页直接再来一局或返回首页。

### 17.2 本地存档

- 刷新页面后最高分不丢失。
- 刷新页面后最近一局成绩不丢失。
- 设置项刷新页面后仍保留。
- 重置本地数据后状态恢复默认。

### 17.3 设置功能

- 可打开设置面板。
- 可切换音效、震动、动画。
- 设置变更会立即生效或被记录。

### 17.4 游戏体验

- 移动端竖屏布局不乱。
- 游戏主路径不被说明文案干扰。
- 结算后的“再来一局”路径明确。

## 18. 对 iOS 上架的价值

完成本文档定义的两步后，Web 版会更适合进入后续 iOS 封装阶段，因为它将具备：

- 更完整的游戏产品结构。
- 更明确的开始与结算闭环。
- 本地持久化能力。
- 用户偏好系统。
- 更接近原生 App 的信息架构。

这会显著降低后续用 `Capacitor` 或其他方案封装为 iOS App 时的产品缺口。

当前 `星块爆破` 已经完成了一个可靠的玩法 MVP。接下来最正确的方向，不是继续加玩法复杂度，而是把它做成一个完整、稳定、可留存、可封装的游戏产品。
